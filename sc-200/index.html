🗺️ The hybrid environment — where each tool fits

Most SC-200 scenarios sit in a hybrid estate: on-premises Active Directory synced to the cloud, endpoints and servers on-prem, and Microsoft 365 + Azure in the cloud. Knowing which product defends which layer is half the exam.

Layer in the environmentDefends it
On-premises domain controllers (AD DS, Kerberos/NTLM, LDAP)Defender for Identity — sensors on the DCs detect recon, lateral movement, DCSync, Golden Ticket
On-prem identities synced to the cloudMicrosoft Entra Connect syncs to Entra ID; Entra ID Protection covers risky cloud sign-ins/users
~1,000 domain-joined Windows endpointsDefender for Endpoint (EDR) — onboard via Group Policy or Intune
On-prem servers (File, SQL, Exchange on Hyper-V) & Azure VMsDefender for Cloud → Defender for Servers (non-Azure servers join via Azure Arc + AMA); Defender for SQL
Microsoft 365 (Exchange Online, SharePoint, Teams, OneDrive)Defender for Office 365 (email/collab) + Defender for Cloud Apps (SaaS/CASB)
Azure IaaS/PaaS resources & postureDefender for Cloud (CSPM + workload Defender plans)
Everything — correlate, hunt, automateMicrosoft Sentinel ingests all sources via data connectors (SIEM + SOAR)

🛡️ Domain 1 — Microsoft Defender XDR ~35–40%

What is Microsoft Defender XDR?

Microsoft Defender XDR (formerly Microsoft 365 Defender) is a unified pre- and post-breach suite that natively coordinates detection, prevention, investigation, and response across endpoints, identities, email, and applications. It correlates alerts and evidence into incidents and is managed from the Microsoft Defender portal (security.microsoft.com).

Alert vs incident: an alert is one detection; an incident is a correlated group of alerts telling the story of an attack across products.
The workloads
  • Defender for Endpoint (MDE) — endpoints (EDR).
  • Defender for Office 365 (MDO) — email & collaboration.
  • Defender for Identity (MDI) — on-premises Active Directory.
  • Defender for Cloud Apps (MDA) — SaaS apps (CASB).
  • Microsoft Entra ID Protection — cloud identity risk.
  • Defender Vulnerability Management — weaknesses & misconfigurations.

Defender for Endpoint (MDE)

Endpoint detection & response with next-gen antivirus, attack surface reduction, vulnerability management, automated investigation, and rich response actions.

Response actions on a device
  • Isolate device — cut network access (keeps Defender connectivity) to stop spread.
  • Restrict app execution — only Microsoft-signed apps may run.
  • Run antivirus scan · Collect investigation package (forensics) · Live response (remote shell).
  • For files: stop and quarantine.
Prevention & configuration
  • Attack Surface Reduction (ASR) rules — block risky behaviours (Office child processes, LSASS credential theft, executable email content). Run in Audit or Block mode.
  • Custom indicators (IoCs) — allow/block file hashes, IPs, URLs/domains, certificates.
  • Defender Vulnerability Management (TVM) — risk-based weakness prioritisation.
  • Onboarding — local script, Group Policy, Intune, Configuration Manager, VDI.
  • RBAC + device groups — scope who can see/act on which devices.

Defender for Office 365 (MDO)

  • Safe Attachments — detonate attachments in a sandbox before delivery.
  • Safe Links — rewrite and scan URLs at time of click.
  • Anti-phishing — spoof & impersonation protection (mailbox intelligence).
  • Zero-hour Auto Purge (ZAP) — retroactively remove already-delivered malicious mail.
  • Threat Explorer — investigate mail threats & take actions (soft-delete).
  • Tenant Allow/Block List — manual allow/block for senders, domains, URLs, files.
  • Preset security policies (Standard/Strict) — Microsoft's recommended baselines.
  • Attack simulation training — run phishing simulations to train users.

Defender for Identity (MDI)

Monitors on-premises Active Directory using sensors installed directly on domain controllers, AD FS, and AD CS servers.

  • Detects reconnaissance (LDAP/DNS enumeration), lateral movement (Pass-the-Hash, Pass-the-Ticket), and domain dominance (Golden Ticket, DCSync).
  • Uses UEBA and feeds signals into Defender XDR incidents.
Don't confuse: Defender for Identity = on-prem AD threats. Entra ID Protection = cloud identity risk (risky sign-ins/users → risk-based Conditional Access).

Defender for Cloud Apps (MDA)

A Cloud Access Security Broker (CASB) for SaaS visibility and control.

  • Cloud Discovery — analyse proxy/firewall/MDE logs to find Shadow IT and app risk.
  • App connectors — API integration for deep activity/file visibility & governance.
  • Conditional Access App Control — reverse proxy for real-time session policies (block downloads, read-only).
  • Policies — activity, file, and anomaly detection (e.g., impossible travel).

Endpoint hardening & XDR operations

  • Tamper protection — stop attackers/malware disabling Defender AV settings.
  • Controlled folder access (anti-ransomware) · Network protection (block bad domains/IPs) · Web content filtering.
  • EDR in block mode — remediate artifacts even behind a third-party AV.
  • Automation level (per device group) — Full auto-remediates; semi levels need approval.
  • Microsoft Secure Score — M365 posture with improvement actions (vs Defender for Cloud's Secure Score for cloud).
  • Threat analytics — campaign/actor reports + your exposure. Action center — approve/track remediations.
  • Alert tuning/suppression to cut benign noise. Unified RBAC for cross-workload permissions.
  • Submissions (report missed phish/false positives) · Attack simulation training.

XDR incidents, response & advanced hunting

  • Incidents — correlated alerts/evidence; assign owners, set status, classify true/false positive.
  • Automated investigation & response (AIR) — auto-investigate and remediate to cut analyst load.
  • Advanced hunting — KQL over a unified schema (DeviceProcessEvents, EmailEvents, IdentityLogonEvents, CloudAppEvents, AlertInfo + AlertEvidence).
  • Custom detection rules — turn a hunting query into a scheduled alert with response actions.
  • Stream XDR incidents into Microsoft Sentinel with the Defender XDR connector.
☁️ Domain 2 — Microsoft Defender for Cloud ~15–20%

What is Microsoft Defender for Cloud?

A Cloud-Native Application Protection Platform (CNAPP) combining CSPM (posture) and CWPP (workload protection) across Azure, on-premises, AWS, and GCP.

CSPM & Secure Score

  • Foundational CSPM is free — continuous assessment producing recommendations.
  • Secure Score — a percentage of your posture; remediating recommendations raises it (many offer Quick Fix).
  • Microsoft Cloud Security Benchmark (MCSB) — the default initiative driving recommendations & score.
  • Asset inventory — all resources with posture, recommendations, and Defender coverage.
  • Multicloud — connect AWS/GCP accounts for unified posture.

Workload protection plans (CWPP)

Paid Defender plans add threat alerts for specific resource types:

PlanProtects / provides
Defender for ServersVMs — MDE integration, JIT VM access, FIM, vulnerability assessment
Defender for StorageMalware scanning + suspicious access on storage accounts
Defender for SQL / DatabasesThreat detection for databases
Defender for ContainersKubernetes/container hardening & runtime protection
Defender for App Service / Key Vault / DNS / Resource Manager / APIsThreat detection for those services (e.g., Resource Manager = suspicious control-plane ops)
  • Just-in-Time (JIT) VM access — keep RDP/SSH closed; open on approved, time-limited request.
  • File Integrity Monitoring (FIM) — watch critical files/registry/OS for changes.
  • Workflow automation — trigger a Logic App on alerts/recommendations (ticket, notify, remediate).
Posture ops: the paid Defender CSPM plan adds attack path analysis, the cloud security explorer, and agentless scanning. Use exemptions to exclude a resource from a recommendation, governance rules to assign owners/due dates, and continuous export to send alerts to Log Analytics/Event Hub. The Azure Monitor Agent (AMA) replaces the legacy MMA agent.

Regulatory compliance

  • Regulatory compliance dashboard — maps your environment to standards (ISO 27001, PCI-DSS, NIST, CIS).
  • Adding standards beyond the default benchmark generally requires an enabled Defender plan.
  • Improve compliance by remediating the recommendations mapped to each control.
  • Send Defender for Cloud alerts to Microsoft Sentinel via its data connector.
🔭 Domain 3 — Microsoft Sentinel ~40–45%

Workspace, architecture & roles

  • Sentinel is a cloud-native SIEM + SOAR built on a Log Analytics workspace; queried with KQL.
  • SIEM = collect/detect/analyse; SOAR = orchestrate/automate/respond.
  • Roles: Sentinel Reader (view), Responder (manage incidents), Contributor (create content), Playbook Operator (run playbooks), Automation Contributor (let Sentinel run playbooks).
  • Commitment tiers reserve daily ingestion volume at a discount vs pay-as-you-go.
Log plans, retention & scale
  • Table plans: Analytics (full features/alerting) · Basic (cheap, high-volume/low-value, limited queries) · Auxiliary.
  • Retention: interactive retention for querying + archive/long-term for cheap compliance storage; query archived data with search jobs / restore.
  • Summary rules aggregate high-volume data to cut cost. Codeless Connector Platform builds connectors without code.
  • Repositories deploy content as code (GitHub/DevOps CI/CD). Azure Lighthouse + workspace() enable cross-workspace/MSSP scenarios.

Data connectors & ingestion

  • Data connectors ingest sources — Microsoft first-party (often no extra ingestion cost), CEF/Syslog via the Azure Monitor Agent (AMA) on a Linux forwarder, REST APIs, and codeless connectors.
  • Data Collection Rules (DCRs) define what the AMA collects (filter to control cost).
  • Content hub — packaged solutions (connector + analytics + workbooks + playbooks) per product.

Analytics rules (detections)

TypeUse
ScheduledRun a KQL query on an interval → alerts/incidents
Near-real-time (NRT)Runs ~every minute for fastest detection
Microsoft SecurityCreate incidents from Microsoft security product alerts
FusionML multistage attack correlation (high-fidelity incidents)
Anomaly / Threat IntelligenceBehavioural anomalies · match logs to TI indicators
  • Entity mapping — map query columns to accounts/hosts/IPs for investigation.
  • Alert grouping — combine related alerts into one incident to cut noise; tune queries to reduce false positives.

Automation (SOAR)

  • Playbooks — automated workflows built on Azure Logic Apps (block user, isolate device, notify Teams). Use the incident trigger for full context.
  • Automation rules — centrally orchestrate incidents: run playbooks, assign owners, change status/severity, tag, suppress.
  • Grant Automation Contributor so automation rules can run a playbook (least privilege).

Hunting, workbooks & UEBA

  • Hunting queries — proactive KQL (mapped to MITRE ATT&CK) for undetected threats; save findings as bookmarks (attach to incidents).
  • Livestream — run a hunt continuously for near-real-time matches.
  • Notebooks (Jupyter/MSTICPy) — advanced code-driven investigation.
  • UEBA — behavioural baselines & anomalies (BehaviorAnalytics table).
  • Workbooks — interactive dashboards. Watchlists — imported reference lists for queries.

Threat intelligence & incident management

  • TI connectors — TAXII feeds or the Upload Indicators API / platforms connector → ThreatIntelligenceIndicator table.
  • TI-map analytics rules match your logs against indicators to raise alerts.
  • Incidents — triage, assign, and close with an accurate classification; explore scope with the investigation graph and entities.

🎯 Exam-day tips

  • Know which product solves which problem — most questions are "which tool/feature?" scenarios.
  • On-prem AD → Defender for Identity; cloud identity risk → Entra ID Protection.
  • "Create incidents from Microsoft alerts" → Microsoft Security rule; "multistage ML" → Fusion; "fastest" → NRT.
  • Automate a response → playbook; orchestrate the incident → automation rule.
  • Be comfortable reading KQL — expect to pick the right operator or query.
  • Watch for NOT, least privilege, and "choose two" multi-response questions.
